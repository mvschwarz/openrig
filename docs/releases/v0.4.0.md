# OpenRig v0.4.0

## Summary

OpenRig v0.4.0 is a token-efficient-defaults + recovery-capstone release.
Five read-commands every agent runs constantly (`rig ps`, `rig whoami`,
`rig queue list`, `rig restore-check`, `rig context`) flip from
firehose-by-default to compact-by-default with an explicit `--full` /
`--all` opt-in; `rig queue list` adopts the docker/kubectl-aligned
read-command grammar (`-a` for history, `-A` for cross-rig breadth,
`-o json|wide` for encoding, `--mine` / `--source` / `--destination`
for scope). `rig seat clear-attention` extends to cover the
restoreOutcome-derived projection staleness class that v0.3.4 shipped
but couldn't reach. `rig scope mission|slice create` now scaffolds
`PROGRESS.md` reliably and ships a `progress` update verb so agents
update progress through the command rather than hand-editing markdown.

Behind those user-visible defaults: the daemon-side payload
source-dedup (`rig ps` `recoveryGuidance` no longer duplicated
per-node, `contextUsage` compact in the list payload) ships at slice 26
so even `--full` and the UI consumers stop paying for it; live agent
terminals replace the read-only 3-second snapshot view (slice 01);
multi-host dogfooding hardens the `--host` flag's cross-host story
(slice 02); the Agent Images library gains a Fork-now affordance + row
metadata (slice 07); attention/activity detection consumes
`agentActivity` truth more richly across the UI (slice 09);
skill-cascade provenance + freshness gets a new read-only `rig skill
audit` surface (slice 10); the identity refactor foundation lands the
common identity types both the daemon and CLI consume (slice 11);
`rig scope` progress rails reconnect the mission/slice tree to
`PROGRESS.md` to the Project UI (slice 15); Codex resume preserves the
launching seat's approval posture so resumed seats can run lifecycle
commands without hitting implicit-deny (slice 17); the active/idle
node-state ring becomes reliable (slice 18); a diagonal ghost in the
topology graph view is fixed at source (slice 21); convergence-fix
P1 (For-You drill captured/live contract) + P2 (daemon test-harness
WebSocket registration) land as the wrap-blocking bug-fix slice 24.

Existing v0.3.4 databases upgrade by running `rig daemon start` on the
new daemon. Migrations are additive.

## Included In This Release

### New top-level CLI verbs + new subcommands

- **`rig skill audit` (slice 10)** — new read-only command + daemon
  surface (`/api/skills/audit`, discovery, mirror-drift detection,
  skills route). Reports skill-cascade provenance findings —
  missing / stale / self-referential / invalid-date / mirror-drift —
  across the canonical → product mirror → hub cwd → installed plugin
  cascade. Prevents false-green CLI success when mirror-drift evidence
  is unavailable (the failure mode the v0.3.4 wrap-gate AC-3 almost
  shipped). Composes with the existing `scripts/mirror-skills.mjs`
  guardrails.
- **`rig scope mission progress` / `rig scope slice progress` (slice 33)** —
  new `progress` verbs that append/set/update entries in `PROGRESS.md`
  deterministically through the command surface rather than hand-edited
  markdown. Writes the canonical structure the OpenRig PROGRESS UI page
  reads.

### Compact-by-default read commands (the token-efficiency family)

- **`rig ps` — compact default + `--full` (slice 25)** — default
  emits a compact TL;DR per node (session, rig, activity state +
  reason, assigned/pending counts; identity-recovery essentials only).
  `--full` returns the complete record (raw byte-equivalent). Same
  breadth as today — all visible nodes still listed; the change is
  projection-trim, not scope-narrowing. `--rig <name>` / `--session
  <sess>` narrow further when wanted. Closes a ~77,000-token
  status-glance incident at root.
- **`rig ps` payload source dedup (slice 26)** — daemon node-list
  payload trims at source: `recoveryGuidance` is no longer serialized
  as near-identical templated prose on every node (relocated to a
  guidance-by-reference map at the top level so the 4 current
  consumers still resolve it); `contextUsage` is a compact summary in
  the list payload (full telemetry remains retrievable per-node via
  whoami / detail query). The ~7 contextUsage UI consumers + 4
  recoveryGuidance consumers migrated without regression. Even
  `--full` and the UI Project view stop paying for the redundant
  per-node blobs.
- **`rig whoami` — compact default + `--full` (slice 27)** — `rig
  whoami` and `rig whoami --json` default to identity-recovery
  essentials only (`identity` / `peers` names + sessionNames / `edges`
  directional kind + to.sessionName / `transcriptPath`). `--full`
  (alias `--verbose`) returns today's complete payload including
  `contextUsage`, `commands`, `peersNote`, `runtimeContext`. Daemon
  skips the `contextUsageStore` lookup and `runtimeContext` build when
  compact is requested. The first command every agent runs on boot +
  every compaction-restore now costs ~192 tokens instead of ~909.
- **`rig queue list` — active-frontier compact default + read-command
  grammar (slices 28 + 32)** — `rig queue list` (no flags) returns
  active states only (`pending` / `in-progress` / `claimed` /
  `blocked` / `handed-off`; not `done` / `canceled`), current-rig
  breadth, compact rows (id, state, source→destination /
  current-owner, closure_reason / closure_target when handed-off /
  blocked, mission, slice, tier / priority, age / updated_at, short
  title, capped tags). The four orthogonal axes are composable:
  - **`-a` / `--all`** — include closed/done history within the
    current breadth (the docker `-a` axis: history).
  - **`-A` / `--all-rigs`** — widen to cross-rig breadth (the
    kubectl `-A` axis: breadth).
  - **`--full`** — add body / chain-of-record / full-tags / transition
    history to whichever scope is selected (the field-breadth axis).
  - **`-o json|wide`** — encoding, compact-by-default. `-o json`
    does NOT imply full body (compact JSON is token-safe + machine-
    parseable); `--full -o json` returns the full JSON.
  - `--mine` returns just the caller's items; `--destination <s>` /
    `--source <s>` for explicit scoping. The bare unscoped firehose
    that aggregated cross-rig + full-history (~64,000 tokens on the
    live host) is retired as a default — opt-in via `-A -a --full`.
- **`rig restore-check` — summary + not-ready default + `--full`
  (slice 29)** — default emits summary counts (total / ready / not-
  ready / error-degraded) plus the not-ready seats only (each compact:
  seat + readiness reason). `--full` / `--json` returns today's
  complete per-seat readiness across the fleet. Daemon skips
  per-seat-detail assembly for ready seats when compact (computes
  verdict, omits detail). `--rig <name>` / `--as <me>` narrow.
  Closes the largest measured bomb (~79,000 tokens) on the
  read-command surface.
- **`rig context` — compact summary default + `--full` (slice 30)** —
  default emits a compact context summary (the few fields the common
  use needs); `--full` / `--json` returns the complete current
  payload. Daemon skips the expensive aggregation when compact. Lower
  leverage than 28/29 but keeps the read-command surface coherent
  compact-by-default after the upgrade.

### Recovery + resilience capstone

- **`rig seat clear-attention` extends to derived projection staleness
  (slice 16)** — v0.3.4 shipped `clear-attention` gating on
  `session.startupStatus` only. First real production use (the host
  cutover's residual `advisor-lead@kernel` projection) found the gap:
  when a seat is `startupStatus=ready` + `sessionStatus=running` but
  carries `restoreOutcome=failed` / `continuityOutcome=failed`, the
  daemon derives `lifecycleState=attention_required` and the verb
  refused with `{ok:false, code:not_in_attention}`. v0.4.0
  `clear-attention` now reaches both classes — the
  startupStatus-driven attention (already shipped) AND the
  restoreOutcome-derived projection — with the same evidence-gated
  audit row and the same operator-attested `--reason <text>` override.
- **Codex Resume Preserves Approval Posture (slice 17)** — a
  resumed Codex seat now comes back with the same permission /
  approval posture as a fresh Codex launch. Previously a resumed
  seat hit "approval required but approvals disabled" implicit-deny
  with no path when running shipped lifecycle commands. Bounded
  bug-fix grounded against the 0.4.0 resilience set; surfaced
  during the GS1 Studio capture.

### UI + topology surfaces

- **Real (Interactive) Terminals (slice 01)** — the per-seat
  terminals in the UI become interactive. Click into a seat, type at
  the agent directly (shared-pane; "hands near the wheel"); the
  read-only 3-second snapshot view is retired for local-host seats.
  Performance constraint met for the rig-altitude "all terminals"
  view (~20–30 panes open simultaneously). Remote-seat terminals
  ride 0.4.1 (need daemon-to-daemon transport).
- **Agent Images Library Polish (slice 07)** — the Agent Images
  library UI gains a `Fork now` action for cloneable images
  (composes the existing add-member endpoint with
  `session_source.mode: "agent_image"`), lightweight row metadata
  (version, token, fork-count, last-used), and nested-failure
  rendering (nested `failed` / `attention_required` outcomes render
  as red failure rather than green success). Fork-now is hidden /
  disabled when the image lacks `sourceCwd`; the confirm path
  requires a runtime-matched sibling carrying `agentRef` + `profile`;
  no fallback cwd is invented.
- **Attention / Activity Detection — Elite Tier (slice 09)** —
  richer consumption of the `agentActivity` truth landed in 0.3.4
  across UI + operator surfaces. Needs-input prominence,
  time-in-state, activity rollups POST-18.
- **Topology Graph-View Ghost Render Fix (slice 21)** — diagonal
  ghost artifact behind the nodes in the dashboard topology /
  graph view is fixed at source. Was present in every graph-view
  capture across the launch-video reviewer kit; the table view was
  already ghost-free. Hero surface across launch videos +
  dashboard.
- **Reliable Active/Idle Node State (slice 18)** — the lit/dark
  activity ring already shipped; this slice closes the
  `DOT→terminalActive` fallback and removes the dead
  `pane_silence_flag` derivation. Flagship video F1's signal layer.

### Scope automation

  PROGRESS UI page parses) plus a convention-correct README with
  proper frontmatter, per
  `conventions/scope-and-versioning/README.md`. Fixes the bug where
  newly-created missions + slices were missing `PROGRESS.md`
  entirely. A `rig scope ... progress` update verb appends / sets /

## User-Facing Changes

### New surfaces (use these)

- **Read-commands compact by default**: `rig ps`, `rig whoami`,
  `rig queue list`, `rig restore-check`, `rig context`. Same
  breadth, much smaller default payload. Use `--full` (or `-A -a
  --full` for queue list) when you need the firehose.
- **`rig queue list` grammar (docker / kubectl aligned)**:
  ```
  rig queue list                      # active, compact, current-rig
  rig queue list -a                   # + closed/done history
  rig queue list -A                   # cross-rig breadth (same as -A in kubectl)
  rig queue list --full               # add body + chain-of-record
  rig queue list -o json              # compact JSON (token-safe)
  rig queue list --full -o json       # full JSON
  rig queue list --mine               # caller's items only
  rig queue list --destination <s>    # destined to <s>
  rig queue list --source <s>         # sourced by <s>
  rig queue show <qitemId>            # full single item (kubectl describe)
  ```
- **`rig skill audit`** — read-only audit of the skill cascade.
  Detects missing / stale / self-referential / invalid-date /
  mirror-drift across the canonical → product mirror → hub cwd
  → installed plugin chain. Prevents false-green CLI success when
  mirror-drift evidence is unavailable.
- **`rig scope ... progress`** — deterministic `PROGRESS.md`
  updates. Append entries, set status, record milestones through
  the command surface; the UI reads the same canonical structure
  the command writes.
- **`rig seat clear-attention` for restoreOutcome-derived
  projection** — the v0.3.4 verb now reaches the second class of
  projection staleness. Same evidence-gated audit row; same
  `--reason <text>` operator-attestation override.

### What to STOP using

- **Stop assuming `rig ps --nodes --json` is the casual status
  check.** Compact default is the casual check; the firehose is
  `--full`. Per-node `recoveryGuidance` near-identical-prose
  duplication is retired; per-node full `contextUsage` is retired
  from the list payload (relocate to single-node / whoami / detail
  queries).
- **Stop using bare `rig queue list` as the cross-rig firehose.**
  Default is now active + compact + current-rig. The cross-rig
  ambient-frontier view is `-A`; cross-rig + history is `-A -a`;
  the full firehose is `-A -a --full -o json`.
- **Stop using `rig whoami --json` to get the heavy payload on
  boot.** Default is compact identity-essentials; `--full` (or
  `--verbose`) returns today's complete payload.
- **Stop using `rig restore-check` as a per-seat-detail fleet
  scan.** Default is summary + not-ready only; `--full` for the
  per-seat detail when actually needed.
- **Stop hand-editing `PROGRESS.md` with markdown to record
  progress.** Use `rig scope ... progress` — UI-valid by
  construction, deterministic, no drift.
- **Stop hand-creating `PROGRESS.md` for new missions / slices.**
  `rig scope mission create` + `rig scope slice create` scaffold
  it now.
- **Stop applying the token-efficiency-boot-guardrail's
  CLI-command prohibitions on hosts running 0.4.0** (the
  prohibitions were host-version workarounds for the bloated
  defaults this release closes). The pack's bounded-local-search
  rule + scope/over-flag discipline GRADUATE to a standing
  convention (`conventions/bounded-local-search-and-flag-scope`)
  and continue to apply host-independently. The
  CLI-command-prohibitions half of the pack RETIRES at
  host-upgrade-to-0.4.0.

### Operator / Setup Notes

- **No new schema-bumping migrations.** Existing v0.3.4 databases
  upgrade by running `rig daemon start` on the new daemon.
- **`rig scope` backfill on existing trees**: missions and slices
  created before 0.4.0 that are missing `PROGRESS.md` can be
  backfilled by running the `rig scope` repair / doctor step (see
  slice 33 IMPLEMENTATION-PRD for the exact verb).
- **Compact JSON is machine-parseable**. The compact-row schema
  for `rig queue list -o json` is stable; downstream parsers
  reading the compact rows don't need migration. Parsers that
  read body / chain-of-record must use `--full` explicitly.
- **`rig skill audit` on the host** — first real production run
  surfaces the current cascade state (missing / stale /
  mirror-drift findings). The lifecycle's cascade-propagation
  pass (canonical `openrig-work/skills/` → product mirror → hub
  cwd → installed plugin) coordinates with the audit's findings
  going forward.

### Known Limitations + carry-forward

- **Plugin-Lineage Drift in `openrig-core`**: the openrig-core
  plugin skill lineage is divergent / stale (see OPR.0.4.1.4 —
  the full plugin re-sync, rides 0.4.1). v0.4.0 ships with this
  drift known; the cascade-sweep this release ran verified the
  canonical + config-mirror layers agents boot from are correct
  / current at the release SHA, and `rig skill audit` (slice 10)
  is the new runtime mechanism to detect future drift. The full
  openrig-core plugin re-sync lands in 0.4.1.
- **`rig ps current-rig default` (slice 34)** + **`rig scope`
  stage / verified verbs (slice 35)** were PUSHED to 0.4.1 per
  founder partition mid-wrap. Their cascade rides 0.4.1's
  durability-close, not this release's.

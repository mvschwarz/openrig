# OpenRig v0.2.0

## Summary

OpenRig v0.2.0 is a major step forward. It ships a durable
daemon-backed coordination layer â€” streams, queues, projects, views,
watchdog supervision, and workflow runtime â€” together with a redesigned
restore experience, an agent-starter registry, cross-host operations,
and a wide set of lifecycle and recovery hardening fixes. This is the
first public release after a long internal hardening cycle, so the
delta from v0.1.12 is large.

## Included In This Release

- A new daemon-owned coordination layer: `rig stream`, `rig queue`,
  `rig project`, `rig view`, `rig watchdog`, and `rig workflow`.
- An agent-starter registry that lets RigSpec members reference named
  startup-priming entries (`starter_ref`) instead of inlining them.
- A new restore-packet command surface for capturing agent
  conversation packets across runtimes.
- New member-level launch source modes: `session_source: { mode:
  rebuild }` and `session_source: { mode: fork }`.
- Cross-host operations: `rig ps`, `rig whoami`, `rig send`, and
  `rig capture` accept `--host` for SSH-based shell-out.
- Substantial lifecycle, recovery, and runtime-truth improvements
  across Claude Code and Codex managed sessions.
- Sixteen new SQLite migrations (021 â†’ 036) to back the new
  coordination, view, watchdog, and workflow tables, plus a few
  observability and config-profile additions.

## User-Facing Changes

### New coordination commands

A new family of `rig` commands provides a durable, daemon-backed
coordination surface:

- `rig stream` â€” emit and query stream items; subscribe via SSE.
- `rig queue` â€” create, claim, hand off, and close queue items.
  Includes `handoff`, `handoff-and-complete`, `update`, `show`,
  `list`, `whoami`, `inbox-drop`, `inbox-absorb`, `outbox-drop`,
  and `outbox-collect`.
- `rig project` â€” group classified work items into durable projects
  with classifier lease/liveness/reclaim semantics.
- `rig view` â€” query built-in and custom views over coordination
  state; consume change events via SSE.
- `rig watchdog` â€” schedule periodic and conditional jobs with
  daemon-native supervision.
- `rig workflow` â€” run multi-step workflows with durable trail
  records and a workflow-keepalive job.

These commands persist their state in the OpenRig daemon's SQLite
database, are observable through the daemon API, and emit
`view.changed` and related SSE events on update.

### New `starter_ref` member field

`rig up` now accepts `starter_ref` on RigSpec members. The daemon
resolves the named entry from the `agent-starters/` registry at
launch and merges its content into the seat's per-cwd guidance file
(`CLAUDE.md` for Claude, `AGENTS.md` for Codex). Inputs containing
credential markers are refused with a redacted refusal envelope.
`starter_ref` is mutually exclusive with inline `session_source` on
the same member, and is rejected upfront in plan-mode for terminal
runtimes and fork compositions.

### New `session_source` modes

Members may now declare:

- `session_source: { mode: rebuild }` â€” rebuild a member from a
  named artifact set on every materialize.
- `session_source: { mode: fork }` â€” fork a new member from an
  existing source on launch.

### Restore-packet command surface

A new `rig restore-packet` family captures restorable conversation
packets across runtimes. It includes parsers and redaction for
Claude and Codex transcripts, schema validation, packet writing, and
a read/validate flow. There is also a `rig restore-check` command
for restore-readiness checks.

### Cross-host operations

`rig ps`, `rig whoami`, `rig send`, and `rig capture` now accept
`--host <ssh-host>` to run against a remote machine. The CLI
shells out over SSH; no long-lived agent or daemon needs to be on
the remote machine ahead of the call.

### Lifecycle, recovery, and runtime-truth improvements

- New lifecycle classifications: `recoverable`, `degraded`, and
  `attention_required`. Codex auth-refusal now surfaces as
  `attention_required` end-to-end.
- `rig up <rigName>` falls back to a manual snapshot when no
  auto-pre-down snapshot exists, so launches no longer fail
  silently in that path.
- Restore now has an explicit attempt id, and runtime-truth
  reconciliation prevents stale state from masquerading as live.
- Codex resume now verifies the pane via a native probe; rig expand
  recovery guidance points to shrink+re-expand instead of trying to
  launch a doomed rig.
- `rig ps --fields` now rejects unknown keys with a per-level
  allow-list. `rig ps` also supports `rigName` aliasing and a
  bounded output mode.

### Agents launched by the daemon now run plain `rig`

Codex seats launched by the daemon previously required an explicit
`OPENRIG_HOME`/`OPENRIG_PORT` env and a non-default binary path on
PATH. Daemon-launched seats now have plain `rig` available, so
agents and operators can use the standard CLI without per-seat env
gymnastics.

### Watchdog help, queue-actor inference, and other small fixes

- `rig watchdog --help` now lists the workflow-keepalive job kind.
- `rig queue` infers the actor from the managed session when no
  explicit actor is provided.
- The CLI bin keeps its executable bit across builds.
- Chatroom wait test fixture is time-stable (no more flaky
  `chatroom wait` test under sanitized environments).

## Operator / Setup Notes

After upgrading the published CLI:

```bash
npm install -g @openrig/cli@0.2.0
rig --version    # expect 0.2.0
```

You will need to restart the daemon to pick up the new schema and
behavior:

```bash
rig daemon stop
rig daemon start
```

The daemon will apply migrations 021-036 to its SQLite database on
first startup. Migrations are forward-only by convention; if you
need to roll back, restore the daemon DB from a snapshot and switch
the daemon binary back to the previous build before restart. Take a
DB snapshot before the first post-upgrade daemon start if you want a
clean rollback path:

```bash
cp ~/.openrig/openrig.sqlite ~/.openrig/openrig.sqlite.pre-v0.2.0.bak
```

If you maintain a managed control-plane environment with live rigs
mid-flight, plan the daemon restart as an upgrade window. The
upgrade is straightforward, but daemon stop/start interrupts active
SSE consumers and any in-progress `rig up` flow.

## Known Limitations

- The new coordination surfaces (`rig stream`, `rig queue`,
  `rig project`, `rig view`, `rig watchdog`, `rig workflow`) are
  daemon-backed. The host daemon must be on v0.2.0 to serve them; an
  older daemon will not expose these commands and may silently
  ignore opt-in fields like `starter_ref` on members.
- `runtime: claude` is not a valid RigSpec runtime spelling; use
  `runtime: claude-code` for Claude managed seats. This is harness/
  spec syntax, not a product limitation, but it has surprised
  operators.
- Watchdog policies and workflow specs are persisted in the daemon
  DB. There is currently no first-class CLI for editing custom
  watchdog policies in bulk; expect more ergonomics in a future
  release.
- The Phase E surfaces (rails, rendering, steering for the
  coordination layer) are intentionally deferred to future product
  lanes (PL-006/PL-007/PL-008) and are not part of v0.2.0.
- Lifecycle classification is more truthful than before, but
  OpenRig still does not have a fully explicit infrastructure-grade
  state machine. Some recovery paths still rely on heuristics
  rather than perfect provider-native primitives.

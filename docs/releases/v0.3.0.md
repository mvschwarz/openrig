# OpenRig v0.3.0

## Summary

OpenRig v0.3.0 turns the 0.2 coordination substrate into a more polished
operator product. It ships the redesigned shell, generic workflow starters,
mission-shaped workspace support, project observability, curated bundled skills,
runtime graphics, context telemetry, and release-hardening work for a cleaner
fresh install.

This release keeps the reusable primitives for queue handoff, workflow runtime,
watchdog supervision, artifact/proof surfaces, and agent coordination, while
the public starter content is intentionally generic.

## Included In This Release

- V1 shell redesign with the vellum visual system, two desktop chrome regions,
  topology graph/table/terminal modes, drawer viewers, Project, For You, Queue,
  Library, Settings, and proof-oriented detail surfaces.
- Generic starter workflow specs:
  - `conveyor` for station-style work where multiple packets can be active
    across the topology at once.
  - `basic-loop` for a one-packet-at-a-time walkthrough flow.
- `conveyor` starter rig and matching agent specs for intake/planning/building
  and review-oriented work.
- Mission-shaped workspace defaults and read-only nested mission/slice indexing.
- Project observability upgrades across For You, Story, Queue, Tests, Project
  rollups, proof screenshots, and qitem drawers.
- Runtime/tool graphics package applied across topology, queue, project metadata,
  proof rows, terminal previews, and agent identity marks.
- Terminal preview improvements, activity indicators, handoff edge animation,
  and context usage telemetry in topology table/terminal views.
- Runtime skill discovery from filesystem skill roots during profile resolution.
- Pre-release CLI/daemon cleanup: bounded transcript rotation, `rig send`
  message envelopes, OpenRig install/init marker naming, and public-release
  terminology cleanup.

## User-Facing Changes

### Redesigned Shell

The UI now uses the V1 shell model with a narrow rail, destination-specific
Explorer region, center workspace, and named-trigger drawer. Topology pages have
graph, table, and terminal modes at one stable URL. Drawer viewers handle queue
items, files, sub-spec previews, terminal previews, and proof assets without
turning every reference click into a full-page navigation.

### Starter Workflows And Rigs

The release includes generic workflow examples rather than project-specific
automation recipes. `conveyor` demonstrates station-based flow with natural
queue backpressure. `basic-loop` demonstrates a simple one-token-at-a-time loop.
The `conveyor` rig is the public workflow starter. `product-team` remains the
more advanced human-operator-driven starter topology.

### Project And Proof Visibility

Project surfaces now distinguish current and archived work, show mission/slice
hierarchy, route mission labels to mission pages, and expose story/tests/proof
rollups. For You and Queue cards hydrate qitem bodies, proof images, audit
outcomes, and next-action state. Tests proof screenshots open in an in-page
viewer.

### Topology And Runtime Visibility

Topology now has a hybrid multi-rig graph, expand/collapse controls, activity
rings, animated handoff edges, runtime/tool marks, context percentage/token
displays, and a terminal fallback view. Agent detail surfaces now use the
canonical live node details view.

### Skills And Starter Guidance

Bundled skills were curated for starter coherence. Runtime skill discovery now
finds valid `SKILL.md` files in runtime and workspace skill roots, with
rig-local resources taking precedence over broader user libraries.

## Operator / Setup Notes

- Fresh workspace defaults are mission-shaped and derive from
  `workspace.root`. Use `rig config init-workspace` to create the default
  workspace structure.
- The repo uses npm workspaces and the tracked root `package-lock.json`.
  `pnpm-lock.yaml` files are not release artifacts.
- Root-level runtime projection files such as `CLAUDE.md` are intentionally not
  tracked in the product repo.
- Deprecated `RIGGED_*` aliases remain only for the original runtime env keys:
  `daemon.port`, `daemon.host`, `db.path`, `transcripts.enabled`, and
  `transcripts.path`. New typed config keys use `OPENRIG_*` only.
- Existing local workflow specs can still be installed from workspace/spec
  locations; product source only bundles the generic starter workflows.

## Known Limitations

- Plugin support is not part of 0.3.0. Plugin-packaged skills and external
  plugin distribution are planned for a later release.
- The public starter workflows are examples, not a complete packaged automation
  methodology.
- Some broad CLI tests are sensitive to local daemon port/config state. Focused
  release-path tests and build/lint gates were used for the final push gate.
